# Testing Guide

## Overview

There are two test suites:

- **Unit tests** — Jest, for isolated logic (no browser, no DB)
- **E2E tests** — Playwright, for full browser flows against a running app + live Supabase instance

---

## Unit Tests (Jest)

### Prerequisites

- Node.js installed
- Dependencies installed: `npm install`

### Run

```bash
npm test
```

No environment variables or running server required.

---

## E2E Tests (Playwright)

### Prerequisites

#### 1. Node.js + dependencies

```bash
npm install
npx playwright install chromium
```

#### 2. Running dev server

The tests hit a real HTTP server. Start it before running tests:

```bash
npm run dev
```

By default tests use `http://localhost:3000`. Override with:

```bash
PLAYWRIGHT_TEST_BASE_URL=http://localhost:3001 npx playwright test
```

#### 3. Supabase instance

Tests create and destroy real data in Supabase. You need a live project (local or hosted).

#### 4. `.env.local` file

Copy `.env.local.example` to `.env.local` and fill in all values:

```bash
cp .env.local.example .env.local
```

Required variables for testing:

| Variable | Where to find it |
|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase project → Settings → API → Project URL |
| `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` | Supabase project → Settings → API → `anon` / `publishable` key |
| `SUPABASE_SERVICE_ROLE_KEY` | Supabase project → Settings → API → `service_role` key |

> `SUPABASE_SERVICE_ROLE_KEY` is **required for E2E tests**. Find it in the project settings in supabase.

### Run

```bash
# All tests
npm run test:e2e

# Interactive UI mode
npm run test:e2e:ui

# Specific project
npx playwright test --project=as-admin

# Single spec file
npx playwright test 07-suspense-bug-condition.spec.ts --project=as-admin

# View last HTML report
npm run test:e2e:report
```

## How E2E Tests Work

### Global setup (`tests-e2e/global-setup.ts`)

Runs once before all tests. It:

1. Creates three ephemeral Supabase auth users (admin, member, other) with unique email suffixes
2. Signs each user in via the browser and saves their auth cookies to `tests-e2e/.auth/`
3. Creates a test club via the app API (using the admin session)
4. Adds the member user to the club with the `Member` role
5. Writes shared test data to `tests-e2e/test-context.json`

### Global teardown (`tests-e2e/global-teardown.ts`)

Runs once after all tests. It deletes all clubs, roles, and auth users created during setup, then removes `test-context.json`.

### Test context (`tests-e2e/test-context.ts`)

Spec files call `loadContext()` to get the shared test data (club ID, user IDs, emails, etc.) written by global setup.

### Auth state files (`tests-e2e/.auth/`)

| File | Used by |
|---|---|
| `admin.json` | `as-admin` project — tests 02, 03, 04, 07, 08 |
| `member.json` | `as-member` project — test 05 |
| `other.json` | `cross-user` project — test 06 |

These files are generated by global setup and are gitignored. Do not commit them.

### Playwright projects

| Project | Auth state | Spec files |
|---|---|---|
| `unauthenticated` | none | `01-*.spec.ts` |
| `as-admin` | `admin.json` | `02-*.spec.ts`, `03-*.spec.ts`, `04-*.spec.ts`, `07-*.spec.ts`, `08-*.spec.ts` |
| `as-member` | `member.json` | `05-*.spec.ts` |
| `cross-user` | `other.json` | `06-*.spec.ts` |

Tests run serially (1 worker) to avoid DB race conditions.

## Build-time Suspense Check (CI)

To catch "Uncached data accessed outside of `<Suspense>`" errors at build time:

```bash
next build --debug-prerender
```

The build must exit with code 0 and produce no `blocking-route` or `Uncached data` strings in output.

## Troubleshooting

**`test-context.json` not found**
Global setup did not run. Always run the full suite with `npx playwright test`, not a bare `node` call to a spec file.

**Login redirect did not occur**
The dev server is not running, or the Supabase credentials in `.env.local` are wrong.

**`SUPABASE_SERVICE_ROLE_KEY` missing**
Add it to `.env.local`. Find it in Supabase → Settings → API → `service_role`.

**Auth files missing (`tests-e2e/.auth/*.json`)**
These are created by global setup. Run the full suite once to generate them, or they will be regenerated automatically on the next run.

**Port conflict**
If `localhost:3000` is taken, start the dev server on another port and set `PLAYWRIGHT_TEST_BASE_URL` accordingly.
